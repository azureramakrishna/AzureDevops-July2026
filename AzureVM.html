Four layers sit between your workload and Azure's physical hardware. Understanding the stack tells you what you control, what Microsoft controls, and where the blast radius of a failure stops.

The shared-responsibility line

        
For IaaS, the line sits between the hypervisor and the guest OS. Microsoft patches the physical host and hypervisor; you patch the OS, your runtime, and your application — including its CVEs. This is the single fact that determines most of an architect's day-two workload on VMs.

      

      

        
Why size is the master setting

        
A VM's size (e.g. Standard_D4s_v5) isn't just "how big" — it sets vCPU count, memory, temp disk, max data disks, max NICs, and max network/storage throughput simultaneously. Changing size later means a reboot, not a re-architecture, which is what makes Azure VMs forgiving to start small with.

      U3
      
Scaling — vertical first, horizontal at production

      
A single VM has a ceiling. Two mechanisms push past it, and they solve different problems.

    


    

      

        

          
        

        
Vertical scaling — resize

        
Stop the VM, change its size (e.g. D2s_v5 → D8s_v5), start it again. More vCPU and RAM on the same instance. Simple, but capped by the largest size in that family and the chosen region, and it always means downtime.

        
Use when: a single workload outgrows its box and brief downtime is acceptable.

      

      

        

          
        

        
Horizontal scaling — VM Scale Sets

        
A Virtual Machine Scale Set manages a fleet of identical VMs behind a load balancer, adding or removing instances based on CPU, memory, queue length, or a schedule. No single-instance downtime — the fleet absorbs load while individual members come and go.

        
Use when: traffic is variable, or you need resilience to a single instance failing.

      

    


    

      
IF avg CPU > 70% for 10 min → add 2 instances

      
IF avg CPU < 25% for 10 min → remove 1 instance

      
Min 2 · Max 20 · across 3 zones

    

  


  
  

    

      U4
      
Storage & disks — three disks, one rule about persistence

      
Every VM has at least an OS disk and a temp disk by default; data disks are added as needed. Only one of these survives everything.

    


    

      

        Virtual machine
        

          OS disk
          Managed disk · persistent · holds the boot volume
        

        

          Temp disk
          Local NVMe/SSD on the host · ephemeral · wiped on host move
        

        

          Data disk(s)
          Managed disk · persistent · attach up to size-defined max
        

      

      

        
Standard HDDBackup, infrequent access

        
Standard SSDWeb servers, light dev/test

        
Premium SSDProduction DBs, low-latency I/O

        
Premium SSD v2 / UltraTunable IOPS independent of size, SAP/HPC

      

    


    

      The one fact that causes outages: the temp disk is not backed by durable storage. If the host fails or the VM is moved, anything written only to the temp disk is gone. Databases, application state, and anything you'd be upset to lose belong on a managed data disk.
    

  


  
  

    

      U5
      
Networking — the VM is one tenant in a virtual network

      
A VM never sits alone. It's a NIC inside a subnet, inside a VNet, governed by rules that exist whether or not you've thought about them yet.

    


    


    

      

        
Network Security Group

        
Stateful firewall rules at the subnet or NIC level. Default rules deny all inbound from the internet — you open only what's needed, by port and source.

      

      

        
Public vs private IP

        
A public IP makes a VM internet-reachable directly — rare in production. Most VMs sit on private IPs behind a load balancer, bastion, or VPN gateway.

      

      

        
Accelerated networking

        
Bypasses the host's software switch via SR-IOV for lower latency and higher throughput — available on most general-purpose and compute-optimized sizes at no extra cost.

      

    

  


  
  

    

      U6
      
Availability — what survives which failure

      
Azure's resilience options map to specific failure domains. Pick based on what you're insuring against, not by reflex.

    


    

      

        
Single VM

        
No redundancy

        
Survives nothing beyond automatic host-level healing. Fine for dev/test; never for a production SLA conversation.

        
No SLA

      

      

        
Availability Set

        
Survives rack failure

        
Spreads VMs across distinct fault domains (power/network) and update domains (host patching groups) inside one datacenter.

        
99.95% SLA

      

      

        
Availability Zones

        
Survives datacenter failure

        
Spreads VMs across physically separate datacenters within a region, each with independent power, cooling, and networking.

        
99.99% SLA

      

      

        
Paired region + Site Recovery

        
Survives region failure

        
Asynchronous replication to a second region for disaster recovery — your last line when an entire region is unreachable.

        
RTO/RPO-defined

      

    

  


  
  

    

      U7
      
Pricing models — the same VM, four different bills

      
Compute cost is a purchasing decision as much as a technical one. The size doesn't change; the commitment does.

    


    

      

        
Pay-as-you-go

        
No commitment

        
Billed per second, on demand. Highest unit price, total flexibility. The default until you understand your steady-state load.

      

      

        
Reserved Instances

        
1 or 3-year term

        
Commit to a VM size/region for a term in exchange for up to ~70% off pay-as-you-go. Right for predictable, always-on workloads.

      

      

        
Spot VMs

        
Interruptible

        
Bid on Azure's unused capacity for steep discounts — Azure can evict your VM with short notice when capacity is needed elsewhere. Right for batch jobs, CI, and stateless workers that tolerate interruption.

      

      

        
Azure Hybrid Benefit

        
License reuse

        
Apply existing on-prem Windows Server or SQL Server licenses with Software Assurance to cut the licensing portion of the VM cost.

      

    

  


  
  

    

      U8
      
Security — defaults that matter on day one

      
Most VM breaches trace back to one of these being skipped, not to anything exotic.

    


    

      

        01
        
Managed identity over secrets

        
Let the VM authenticate to other Azure services via a system-assigned identity instead of embedding connection strings or keys in config files.

      

      

        02
        
Just-in-time VM access

        
Lock down management ports (RDP/SSH) by default; open them only on request, for a limited window, via Microsoft Defender for Cloud.

      

      

        03
        
Disk encryption at rest

        
Azure Disk Encryption (BitLocker/DM-Crypt) or platform-managed encryption protects data disks; bring your own keys via Key Vault when compliance requires it.

      

      

        04
        
Patch management

        
Automatic guest patching or Azure Update Manager closes the OS-level gap that sits squarely on your side of the shared-responsibility line.

      

      

        05
        
Bastion over public RDP/SSH

        
Azure Bastion provides browser-based RDP/SSH over TLS without ever exposing a public IP on the VM itself.

      

      

        06
        
NSG flow logs

        
Capture accepted/denied traffic at the network layer for forensics — the first thing an incident review asks for and the first thing teams forget to enable.

Decision guide — VM, or something else?

      
A VM is the right answer when you need OS-level control. It's the wrong answer when a managed service would do the same job with less to operate.

    


    

      

        
If you need…Reach forNot a VM, because…

        
          
Full OS control, custom drivers, legacy app dependenciesVirtual Machine—

          
Run containers without managing orchestrationAzure Container Apps / ACIYou'd be patching an OS just to run Docker

          
Kubernetes at scale, custom networking/CNIAzure Kubernetes ServiceAKS manages the control plane and node patching for you

          
Run code on events, no idle costAzure FunctionsPaying for a VM to sit idle between triggers is waste

          
Managed relational/NoSQL databaseAzure SQL DB / Cosmos DBYou'd own backup, patching, and HA logic the PaaS already solved

          
Static site or simple web appAzure App Service / Static Web AppsNo OS-level need; PaaS deploy is faster and cheaper to run

        
      

    


    

      
The architect's question is never "can a VM do this" — it almost always can. It's "does this workload need the control a VM gives you, badly enough to also own everything that control costs."
