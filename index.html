01 · Core Services

Four services, one account

A storage account is the billing and security boundary. Inside it, four distinct data services solve four distinct problems — picking the wrong one is the most common Azure Storage architecture mistake.

B

Blob Storage

Object storage for unstructured data — images, video, backups, logs, data lake files. Three blob types: block blobs (general files, uploaded in chunks), append blobs (optimized for sequential writes like logs), and page blobs (random read/write — what Azure VM disks are built on).

REST · SDK · Data Lake Gen2
F

Azure Files

Fully managed SMB and NFS file shares. Mount it on Windows, Linux, or macOS exactly like a network drive. The default answer when "lift and shift" means moving an on-prem file server, or when multiple VMs need a shared filesystem.

SMB 3.0 · NFS 4.1 · Azure File Sync
Q

Queue Storage

Simple, durable async messaging between application components. A web app enqueues a job; a worker role dequeues and processes it. At-least-once delivery — your consumer must be idempotent or design around duplicate processing.

REST · 64KB msg · 7-day TTL max
T

Table Storage

NoSQL key-value store using a PartitionKey/RowKey schema. Schemaless, fast, cheap. Microsoft now recommends new workloads use Cosmos DB's Table API, which is wire-compatible but adds global distribution and stronger SLAs.

NoSQL · PartitionKey/RowKey
Choosing → Need a network drive multiple VMs mount? Files. Storing app-generated objects or images? Blob. Decoupling a producer from a consumer? Queue. Need a cheap NoSQL lookup table and don't need global distribution? Table — otherwise reach for Cosmos DB.
02 · Durability

Redundancy & replication geography

Every storage account replicates your data automatically — the question is how far the copies travel, and what happens if a datacenter, a zone, or an entire region goes dark.

LRS
Locally Redundant
3 copies, one datacenter, one region. Cheapest option — but a datacenter-level disaster takes your data with it.
11 nines / year
ZRS
Zone Redundant
3 copies across separate availability zones in one region. Survives a full zone outage with no data loss.
12 nines / year
GRS
Geo Redundant
LRS in the primary region, plus 3 more async-replicated copies in a paired secondary region hundreds of km away.
16 nines / year
GZRS
Geo-Zone Redundant
ZRS in the primary region plus geo-replication to a secondary region. The strongest standard option — zone and region failure protection together.
16 nines / year
Append -RA to GRS or GZRS (RA-GRS, RA-GZRS) to get read access to the secondary region even while the primary is healthy — useful for read-heavy global apps, not just disaster recovery. Without the RA prefix, the secondary copy exists but isn't readable until Microsoft declares a failover.
Exam / Interview angle → Geo-replication is asynchronous. During a regional outage, recently written data may not have made it to the secondary yet — your Recovery Point Objective with GRS is typically measured in minutes, not zero. If you need synchronous, zero-data-loss cross-zone durability, that's what ZRS gives you within a single region.
03 · Cost Optimization

Access tiers

Blob storage pricing trades storage cost against access cost and retrieval latency. Pick the tier that matches how often the data is actually read — not how important it feels.

Hot
Optimized for data accessed frequently. Highest storage cost, lowest access cost. Default tier for new accounts.
ms latency
Cool
Data accessed infrequently, stored for at least 30 days. Lower storage cost, higher per-access cost than Hot.
ms latency
Cold
Rarely accessed data, stored for at least 90 days. Even lower storage cost than Cool, higher retrieval cost.
ms latency
Archive
Offline tier for data rarely accessed, stored at least 180 days. Cheapest storage by far — but blobs must be rehydrated before reading.
hours to retrieve
Gotcha → Archive tier blobs aren't instantly readable. Rehydrating moves a blob back to Hot or Cool, taking up to 15 hours on standard priority (or under an hour on high priority, at extra cost). Architect around this latency — Archive is for compliance retention and cold backups, not anything in a live request path.
Tiers can be set per-blob and changed any time, or automated entirely with lifecycle management policies (see below). Early deletion from Cool, Cold, or Archive before the minimum retention window incurs an early-deletion penalty — factor that into any aggressive auto-tiering policy.
04 · Security

Security, layer by layer

Azure Storage security stacks from the network edge down to the individual request — each layer answers a different question about who can do what.

Network rules & Private EndpointsPerimeter

Restrict the storage account to specific VNets, IP ranges, or force all traffic through a Private Endpoint with a private IP — removing it from the public internet entirely.

Microsoft Entra ID (RBAC)Identity

The recommended authorization model. Assign roles like Storage Blob Data Reader or Storage Blob Data Contributor to users, groups, or managed identities — no secrets to rotate or leak.

Shared Access Signatures (SAS)Delegated access

Time-bound, permission-scoped URLs for granting temporary access without sharing account keys — e.g. a browser uploading directly to a container. Always set the shortest practical expiry.

Account KeysRoot credential

Full control over the entire account. Treat exactly like a root password — store in Key Vault, never in code, and rotate regularly. Disable key-based auth entirely on accounts that don't need it.

Encryption at RestAlways on

Storage Service Encryption (SSE) encrypts every byte automatically with AES-256, with no way to disable it. Use Microsoft-managed keys by default, or customer-managed keys in Key Vault for compliance needs.

Encryption in TransitAlways on

"Secure transfer required" enforces HTTPS for REST calls and SMB 3.0 encryption for Azure Files, rejecting any unencrypted connection attempt outright.

Best practice → The modern recommended posture: disable shared key (account key) access, require Entra ID RBAC for everything, route traffic through Private Endpoints, and issue user-delegation SAS tokens (which are themselves backed by Entra ID, not account keys) for any delegated/temporary access.
05 · Networking

Network access paths

Three distinct ways traffic reaches a storage account — each with a different security and latency profile.

PathHow it worksUse when
Public endpoint Default. Account reachable over the internet, optionally locked down with firewall IP/VNet rules. Quick prototyping, or public content like a static website / CDN origin.
Service Endpoint Extends a VNet's identity to the storage account over the Azure backbone — traffic never leaves Microsoft's network, but the account still has a public IP. Simple VNet-restricted access without the cost of Private Link.
Private Endpoint Provisions a NIC with a private IP inside your VNet, mapped via Private Link directly to the storage account. No public IP exposure at all. Production workloads, compliance requirements, zero public attack surface.
Common misconfiguration → Enabling a Private Endpoint does not automatically disable the public endpoint — you must explicitly set "Public network access" to disabled, or the storage account remains reachable both ways. Many security audits flag exactly this gap.
06 · Automation

Lifecycle management

Rule-based policies that move or delete blobs automatically based on age — turning manual tiering into a configure-once-and-forget cost control.

lifecycle-policy.jsonblob → cool → archive → delete
// Move to Cool after 30 days of no access,
// Archive after 90, delete after 365
{
  "rules": [{
    "name": "auto-tier-old-logs",
    "type": "Lifecycle",
    "definition": {
      "filters": { "blobTypes": ["blockBlob"] },
      "actions": {
        "baseBlob": {
          "tierToCool": { "daysAfterModificationGreaterThan": 30 },
          "tierToArchive": { "daysAfterModificationGreaterThan": 90 },
          "delete": { "daysAfterModificationGreaterThan": 365 }
        }
      }
    }
  }]
}
Policies can target by container prefix, blob name pattern, or last-access time (when access-tracking is enabled) — not just modification date. Combine with soft delete and versioning for a complete data-protection story: lifecycle policies handle cost, soft delete and versioning handle accidental loss.
07 · Scalability

Performance & scale limits

Storage accounts have published scalability targets — hitting them shows up as HTTP 503 throttling, not a graceful slowdown. Design partition keys and request patterns around these limits up front.

⚡

Standard vs Premium

Standard accounts use magnetic/SSD-backed storage at lower cost. Premium accounts run on SSD exclusively, offering consistent low latency — the default choice for high-IOPS page blob workloads like VM disks.

∥

Partitioning

Blob storage automatically load-balances by URL path. Hot-partition bottlenecks happen when traffic concentrates on sequential or predictable blob name prefixes — randomize prefixes for high-throughput write patterns.

↻

Throttling & retry

Exceeding request-rate or bandwidth limits returns HTTP 503. The Azure SDKs implement exponential backoff with jitter by default — never disable client-side retry logic in production.

08 · Architecture

Common patterns built on Storage

Storage rarely stands alone — these are the recurring shapes it takes inside larger Azure architectures.

Static website hosting

Browser → Azure CDN / Front Door → $web container (Blob Static Website)

Enable the static website feature on a Blob Storage account to serve HTML/CSS/JS directly from the $web container — no VM, no App Service. Pair with Azure CDN or Front Door for global caching and a custom domain with TLS.

Event-driven processing pipeline

Blob upload → Event Grid → Azure Function → processed output → Blob / Queue

A blob upload fires an Event Grid notification, triggering a Function App to resize an image, parse a file, or kick off a downstream workflow — the backbone of most serverless data-processing pipelines on Azure.

Hybrid file share with Azure File Sync

On-prem Windows Server (cache) ⇄ Storage Sync Service ⇄ Azure Files (cloud tier)

Keeps a local Windows Server as a fast cache while the authoritative copy and cold data live in Azure Files — classic for branch-office file servers being migrated to the cloud without a disruptive cutover.

Data lake for analytics

Ingest (Data Factory) → ADLS Gen2 (hierarchical namespace) → Synapse / Databricks

Blob storage with hierarchical namespace enabled becomes Azure Data Lake Storage Gen2 — true directory semantics and POSIX-like ACLs, purpose-built as the ingestion and storage layer for big-data analytics engines.

09 · Observability

Monitoring & diagnostics

Storage accounts emit rich telemetry by default — the work is wiring it to somewhere you'll actually look during an incident.

ToolGives you
Azure Monitor metricsTransactions, ingress/egress, latency, availability — near-real-time, platform-level metrics with no setup.
Storage Analytics loggingDetailed per-request logs: who, what operation, success/failure, latency — written to a $logs container.
Diagnostic settings → Log AnalyticsStream metrics and logs into a Log Analytics workspace for KQL queries, alerting, and dashboards.
Azure AdvisorAutomated recommendations — flags accounts missing redundancy upgrades, unused soft-delete, or cost-saving tier moves.
Microsoft Defender for StorageThreat detection — flags anomalous access patterns, malware uploads, and unusual data exfiltration.
Operational tip → Set alerts on availability dropping below 99.9% and on throttling errors (HTTP 503) — these surface real production impact long before someone files a support ticket.
10 · Cost

What actually drives the bill

Storage pricing is multi-dimensional — easy to optimize one axis and accidentally blow up another.

$

Capacity

Per-GB/month, varies by tier and redundancy level. The most visible but often not the largest cost.

↑

Operations

Per-10,000-transaction pricing on reads, writes, and list operations — chatty applications can rack this up fast.

↓

Egress

Outbound data transfer is billed; inbound is free. Geo-replication and CDN origin pulls both count as egress.

⏱

Early deletion

Penalty for deleting/moving data out of Cool, Cold, or Archive before the minimum retention period.

Real-world trap → Archive tier is cheap to store but expensive in two hidden ways: rehydration time (hours) and a per-GB retrieval fee on top of the rehydration. A workload that "just checks Archive sometimes" can cost more in retrieval fees than it would have cost to leave the data in Cool the whole time. Model actual access patterns before committing to a tier.
