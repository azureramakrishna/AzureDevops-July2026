Azure Load Balancer, end to end

01

Two types of Load Balancer

Defined by where the frontend IP lives — internet-facing or private-network-facing. A lab or production topology can use either or both together.

Public Public Load Balancer

Frontend has a public IP address. Maps inbound internet traffic to VMs, and also provides outbound internet connectivity for the backend pool via SNAT.

  • Used for internet-facing web tiers
  • Load balances traffic from the internet to VMs
  • Provides outbound-to-internet translation (SNAT)
  • Typical scenario: two web VMs serving a public site

Internal Internal (Private) Load Balancer

Frontend uses a private IP from the VNet only. Distributes traffic among VMs inside a VNet, or reachable via VPN/ExpressRoute — never exposed to the internet.

  • Used for multi-tier apps (web tier → app tier)
  • Frontend IP is a private address (e.g. 10.0.2.10)
  • No direct internet exposure
  • Typical scenario: database or internal API tier

03

Core building blocks

Every Load Balancer — public or internal — is assembled from the same six components.

1 · Frontend IP configuration

The IP address (public or private) clients connect to. A single LB can host multiple frontends.

Public: 20.x.x.x | Internal: 10.0.1.100

2 · Backend address pool

The set of VMs or VM Scale Set instances that will actually receive the traffic.

e.g. VM-1 (10.0.1.4), VM-2 (10.0.1.5)

3 · Health probes

Periodic checks (TCP/HTTP/HTTPS) against each backend instance. Unhealthy instances are automatically removed from rotation.

Interval: 5–15s | Unhealthy threshold: 2 probes

4 · Load balancing rules

Maps a frontend IP+port to a backend pool+port, referencing a health probe and a distribution mode.

Frontend :80 → Backend pool :80

5 · Inbound NAT rules

Forwards traffic from a specific frontend port directly to a specific single VM — commonly used for RDP/SSH management access.

Frontend :50001 → VM-1 :3389 (RDP)

6 · Outbound rules

Explicitly defines how backend pool instances reach the internet (SNAT), giving control over port allocation per instance.

Standard SKU only — explicit config required
04

Traffic distribution & session persistence

Azure Load Balancer uses a 5-tuple hash by default — no configuration needed for basic round-robin-like distribution.

5-tuple hash (default)

A hash is calculated from: source IP, source port, destination IP, destination port, and protocol type. Same 5-tuple → same backend instance for the life of a flow.

Session persistence modes

  • None — pure 5-tuple hash (default, best distribution)
  • Client IP — 2-tuple hash (source IP, destination IP)
  • Client IP and protocol — 3-tuple hash

Lab architecture — 2 VMs behind one Load Balancer

The exact topology this reference is built for: a Standard Public Load Balancer distributing HTTP traffic across two VMs in the same availability set / zone.

# 1. Resource group + VNet/subnet already created; then: az network public-ip create --resource-group lab-rg --name lb-pip --sku Standard --zone 1 2 3 # 2. Create the Standard Load Balancer az network lb create --resource-group lab-rg --name lab-lb --sku Standard \ --public-ip-address lb-pip --frontend-ip-name lb-frontend --backend-pool-name lb-backend-pool # 3. Health probe on port 80 az network lb probe create --resource-group lab-rg --lb-name lab-lb \ --name health-probe --protocol tcp --port 80 --interval 5 --threshold 2 # 4. Load balancing rule: frontend :80 -> backend pool :80 az network lb rule create --resource-group lab-rg --lb-name lab-lb \ --name http-rule --protocol tcp --frontend-port 80 --backend-port 80 \ --frontend-ip-name lb-frontend --backend-pool-name lb-backend-pool --probe-name health-probe # 5. Add each VM's NIC IP config to the backend pool az network nic ip-config address-pool add --resource-group lab-rg \ --nic-name vm1-nic --ip-config-name ipconfig1 \ --lb-name lab-lb --address-pool lb-backend-pool
07

Quick verification & testing commands

Verify from Azure CLI

  • az network lb show — view LB configuration
  • az network lb address-pool list — confirm backend pool members
  • az network lb probe list — check probe health config
  • az network lb rule list — confirm rule mappings

Verify from the client side

  • Browse to http://<lb-public-ip> repeatedly — responses should alternate between VM-1 and VM-2
  • Stop the web service on one VM — traffic should shift entirely to the healthy VM within one probe interval
  • Use curl -s http://<lb-public-ip> in a loop to observe distribution in real time
