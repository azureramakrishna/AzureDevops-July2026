Terraform provisions the VM whose kernel does all the real work. This reference covers the Linux underneath your entire toolchain — the filesystem, permissions model, process lifecycle, networking stack, and the commands you'll actually run at 2am when something's on fire.

01 · Filesystem

The Filesystem Hierarchy Standard

Everything in Linux is a file — sockets, devices, processes, even kernel parameters. Knowing where things live is the difference between debugging in 10 seconds and 10 minutes.

/etc— system-wide config files (nginx.conf, hosts, fstab, systemd units)
/var— variable data: logs (/var/log), caches, container state
/opt— third-party / vendor software (Docker Desktop, JDK, custom agents)
/usr— user-land binaries and libraries (/usr/bin, /usr/lib)
/bin,/sbin— essential binaries needed before /usr is mounted (now often symlinked into /usr)
/proc— virtual filesystem exposing kernel & process info in real time, not on disk
/sys— virtual filesystem exposing kernel objects, devices, and cgroups
/home— per-user personal directories
/tmp— temporary files, usually cleared on reboot
/dev— device files (disks, terminals, /dev/null, /dev/random)
/root— home directory for the root user
WHY IT MATTERS → /proc and /sys aren't real disk files — they're windows into the running kernel. cat /proc/cpuinfo, cat /proc/meminfo, and ls /sys/fs/cgroup are how tools like docker stats and Kubernetes' kubelet actually get their metrics. There's no API call happening — it's a file read.
Config vs Runtime

/etc vs /var

/etc holds the config you wrote and version-control. /var holds what the system generates while running — and what fills up your disk at 3am.

Disk vs Virtual

/proc and /sys aren't disk

They consume zero disk space and reset on reboot. df won't show real usage for them — they're generated on read.

01 · Filesystem

Permissions & Ownership

Every file has an owner, a group, and three permission triads. Get this wrong on a Dockerfile COPY or a Kubernetes volumeMount and you'll spend an hour staring at "Permission denied."

rwx 7 owner
r-x 5 group
r-- 4 others

-rwxr-xr--  →  chmod 754 file.sh

Read (r=4) lets you view file contents or list a directory. Write (w=2) lets you modify a file or add/remove entries in a directory. Execute (x=1) runs a file as a program, or lets you cd into a directory. Add the numbers per triad to get the chmod digit.
chmod[mode] [file]
permissions

Change file mode bits. Use octal for absolute sets, or symbolic for relative tweaks.

$ chmod 644 deploy.yaml
# owner: rw-, group: r--, others: r-- — standard config file
$ chmod 755 entrypoint.sh
# owner: rwx, group: r-x, others: r-x — executable script
$ chmod +x ./deploy.sh
# add execute bit for all, keep everything else unchanged
$ chmod -R 750 /opt/app
# recursive: apply to directory and everything inside it
chown[user]:[group] [file]
ownership

Change file owner and/or group. The classic fix for container volume permission mismatches.

$ chown www-data:www-data /var/www/html -R
# nginx worker process needs to own the files it serves
$ chown 1000:1000 /data
# match container's non-root UID:GID to a mounted host volume
DEVOPS GOTCHA → A bind-mounted volume keeps the host's UID/GID. If your container runs as UID 1000 but the host directory is owned by UID 1001, you get permission denied even though both "look like" the right user. Always check with ls -n to see raw UIDs, not names.
umaskDefault permission mask subtracted from new files (022 → new files get 644, dirs get 755)
setuid (4000)Run binary as the file owner, not the invoking user — e.g. /usr/bin/passwd
setgid (2000)New files in a directory inherit the directory's group — useful for shared team dirs
sticky bit (1000)Only the file owner can delete their own files in a shared dir — e.g. /tmp
02 · Processes

Process Management

Every container is a process tree. Understanding PIDs, signals, and how a parent reaps a child is what's actually happening inside docker stop and a Kubernetes pod eviction.

PID 1 systemd (init — parent of everything)
└─ PID 842 containerd-shim
└─ PID 851 nginx: master process
├─ PID 852 nginx: worker process
└─ PID 853 nginx: worker process
PID 1 is special: it's the first process the kernel starts and the ultimate parent of every other process. If a process's parent dies before it does, PID 1 "adopts" it — this is why a misconfigured container entrypoint that isn't PID-1-aware can leave zombie processes accumulating.
psaux / ps -ef
inspect

Snapshot of running processes. aux is BSD style, -ef is UNIX style — both are everywhere, pick one and learn its columns.

$ ps aux | grep nginx
root 851 0.0 0.1 55 nginx: master process
www 852 0.1 0.4 102 nginx: worker process
$ ps -ef --forest
# --forest renders the actual parent/child tree visually
%CPU / %MEMShare of CPU time and resident memory the process is using right now
STATS=sleeping, R=running, Z=zombie, D=uninterruptible sleep (often disk I/O wait)
top / htop
live monitor

Real-time, continuously refreshing view of CPU, memory, and per-process load. htop is the friendlier, color-coded upgrade — install it, don't fight plain top.

kill / pkill
signals

kill doesn't forcibly destroy a process by default — it sends a signal, and the process decides how to react. This single fact explains why docker stop can hang and then suddenly succeed.

$ kill -15 4521
# SIGTERM — "please shut down gracefully." App can catch this and clean up.
$ kill -9 4521
# SIGKILL — immediate, uncatchable termination. Last resort. No cleanup runs.
$ pkill -f "node server.js"
# kill by matching the full command line instead of a PID
WHY IT MATTERS → docker stop sends SIGTERM, waits a grace period (default 10s), then sends SIGKILL if the process hasn't exited. The same pattern is true of kubectl delete pod and its terminationGracePeriodSeconds. If your app doesn't handle SIGTERM, every rolling deploy is a small forced kill.

foreground vs background

  • command & — run in background, keep shell free
  • jobs — list background jobs in this shell
  • fg %1 / bg %1 — bring job to fore/background
  • nohup cmd & — survive shell logout

nice & renice

  • nice -n 10 cmd — start with lower CPU priority
  • renice -n -5 -p 4521 — raise priority of running process
  • Range: -20 (highest priority) to 19 (lowest)
  • Only root can set negative niceness
02 · Processes

systemd & Service Management

systemd is PID 1 on virtually every modern distro. It starts your services, restarts them on crash, and is the reason a misbehaving app doesn't take the whole VM down.

systemctl
service control
$ systemctl status docker
● docker.service — active (running) since Mon 09:14:02
$ systemctl restart nginx
$ systemctl enable kubelet
# symlinks the unit so it starts automatically on boot
$ systemctl daemon-reload
# re-read unit files after you edit a .service file by hand
commanddoes
systemctl start/stop NAMEStart or stop a service right now (no boot-time effect)
systemctl enable/disable NAMEToggle whether it auto-starts at boot
systemctl is-active NAMEQuick boolean check — perfect for health-check scripts
journalctl -u NAME -fTail live logs for a specific unit
systemctl list-units --failedShow every unit that failed to start — first stop after a bad boot
INTERVIEW ANGLE → "Why does Docker need systemd on the host but not inside the container?" Because the host needs an init system to supervise the dockerd daemon and restart it on crash; the container itself usually runs a single foreground process as PID 1, with no init system needed unless you're running multiple processes (in which case tools like tini or s6-overlay fill that role).

03 · Navigation

Text Processing & Pipes

Logs are text. Configs are text. The Linux philosophy is small tools, chained with pipes — this is the actual skill tested in most "debug this log" interview rounds.

grep
search
$ grep -i "error" app.log
# case-insensitive match
$ grep -rn "TODO" ./src
# recursive, with line numbers
$ grep -v "DEBUG" app.log | grep -c "ERROR"
# exclude debug lines, then count remaining error lines
$ grep -B2 -A5 "OOMKilled" syslog
# show 2 lines before and 5 lines after each match for context
awk / sed
transform

awk thinks in columns. sed thinks in find-and-replace, line by line.

$ awk '{print $1, $9}' access.log
# print column 1 (IP) and column 9 (status code) — nginx log format
$ awk -F: '{print $1}' /etc/passwd
# custom field separator: list usernames
$ sed -i 's/staging/production/g' config.yaml
# in-place find-and-replace across the whole file
$ sed -n '10,20p' file.txt
# print only lines 10 through 20
commanddoes
cat file | tail -f -n 100Follow a log file live, starting from the last 100 lines
wc -l fileCount lines — fast sanity check on log volume
sort file.txt | uniq -c | sort -rnClassic frequency count: most common lines first
cut -d',' -f2,4 data.csvExtract specific columns by delimiter
xargsBuild and run commands from piped input — e.g. find . -name "*.tmp" | xargs rm
jq '.status' response.jsonParse and query JSON — essential for API/CLI output in pipelines
04 · System

Disk & Storage

"The disk is full" is one of the most common production incidents — and almost always caused by logs, container layers, or orphaned volumes nobody cleaned up.

df -h
disk usage

Filesystem-level usage. The first command when a node reports disk pressure.

$ df -h
Filesystem Size Used Avail Use% Mounted on
/dev/sda1 50G 46G 4G 92% /
overlay 50G 46G 4G 92% /var/lib/docker
du -sh */
directory usage

Drill down to find which directory is actually consuming the space df reported.

$ du -sh /var/log/* | sort -rh | head -10
# top 10 largest items under /var/log, biggest first
$ docker system df
# Docker's own breakdown: images, containers, volumes, build cache
$ docker system prune -a --volumes
# reclaim space — removes unused images, networks, and volumes
commanddoes
lsblkList block devices in a tree — disks, partitions, mounted filesystems
mount / umountAttach or detach a filesystem to a directory path
fdisk -lList partition tables on all disks (needs sudo)
mkfs.ext4 /dev/sdb1Format a partition with the ext4 filesystem
fstrim -avTell SSDs to reclaim unused blocks — common on cloud-attached volumes
cat /etc/fstabMounts that should persist across reboot
DEVOPS GOTCHA → A deleted file doesn't free space if a process still has it open. lsof | grep deleted finds these phantom file handles — classic cause of "I deleted the logs but df still shows full disk."
04 · System

Users, Groups & sudo

Least-privilege access starts here — and it's exactly what container security scanners flag when an image runs as root unnecessarily.

commanddoes
useradd -m -s /bin/bash deployCreate a user with a home directory and bash as the default shell
usermod -aG docker deployAdd an existing user to a group (-a appends, doesn't overwrite existing groups)
passwd deploySet or change a user's password
id deployShow UID, GID, and all group memberships for a user
su - deploySwitch user, loading their full login environment
sudo -lList what the current user is permitted to run as root
visudoSafely edit /etc/sudoers with syntax checking before saving
whoami / who / wCurrent user / logged-in users / logged-in users + what they're doing
INTERVIEW ANGLE → "Why shouldn't a Dockerfile run as root?" If an attacker breaks out of the app process, root inside the container often maps to dangerous host capabilities, especially with privileged mode or shared namespaces. The fix: USER appuser in the Dockerfile plus a matching non-root UID, the same principle as usermod -aG instead of handing out root.
05 · Networking

Network Diagnostics

"It works locally but not in the cluster" is a networking problem nine times out of ten. These commands narrow down which layer is actually broken.

ip a / ip route
interfaces

The modern replacement for ifconfig and route. Shows interfaces, IPs, and routing table.

$ ip a show eth0
inet 10.0.4.12/24 scope global eth0
$ ip route
default via 10.0.4.1 dev eth0
commanddoes
ping -c 4 hostLayer 3 reachability check — is the host even alive?
curl -v https://api.internal/healthLayer 7 check — full request/response with headers, TLS handshake details
curl -o /dev/null -s -w "%{http_code}\n" urlPrint only the HTTP status code — perfect for health-check scripts
telnet host 443 / nc -zv host 443Raw TCP port check — is something even listening, regardless of protocol?
traceroute hostShow every network hop between you and the destination
dig domain.com / nslookup domain.comDNS resolution — first suspect when "it can't find the host"
ss -tulnpList listening ports and the process bound to each (modern netstat)
mtr hostCombines ping + traceroute as a live, continuously updating view
DEBUGGING LADDER → When a service is unreachable, check in order: 1) ping — is the host up? 2) nc -zv host port — is the port open? 3) curl -v — does the app actually respond? Each layer rules out a different failure class before you go chasing application code.
05 · Networking

iptables & Packet Filtering

Docker and Kubernetes both manipulate iptables rules behind the scenes to implement container networking and Service routing. Understanding the chains demystifies a lot of "magic."

Packets pass through ordered chains. The three built-in chains in the filter table: INPUT (traffic destined for this host), OUTPUT (traffic leaving this host), and FORWARD (traffic passing through this host to somewhere else — exactly what container bridge networking relies on).
iptables
firewall
$ iptables -L -n -v
# list all rules in the filter table, numeric, with packet counters
$ iptables -A INPUT -p tcp --dport 22 -j ACCEPT
# Append rule: allow inbound SSH
$ iptables -A INPUT -p tcp --dport 8080 -s 10.0.0.0/8 -j ACCEPT
# allow port 8080 only from the internal 10.0.0.0/8 range
$ iptables -P INPUT DROP
# set default policy to DROP — deny-by-default, only explicit ACCEPTs pass
$ iptables -t nat -L -n
# inspect the NAT table — this is where Docker publishes container ports
WHY IT MATTERS → When you run docker run -p 8080:80, Docker isn't doing anything exotic — it inserts a DNAT rule into iptables' nat table that rewrites packets headed to host port 8080 toward the container's internal IP on port 80. iptables -t nat -L DOCKER -n shows it directly.
-AAppend a rule to the end of a chain
-IInsert a rule at a specific position (often the top, for priority)
-DDelete a matching rule
-j ACCEPT/DROP/REJECTThe action to take — DROP silently discards, REJECT replies with an error
-s / -dMatch by source / destination IP or CIDR range
06 · Containers

Linux Namespaces

Containers aren't a kernel feature — they're a combination of two unrelated kernel features. This is one, and the most-asked "what actually is a container" interview answer.

A namespace wraps a global system resource so that processes inside it see their own isolated instance of that resource. Docker creates a new set of namespaces per container; that isolation is the container, from the kernel's point of view.
PID namespace

Process isolation

Inside the container, your app is PID 1. On the host, it's some high-numbered PID. Container can't see or signal host processes.

NET namespace

Network isolation

Own network interfaces, IP, routing table, port space. Why two containers can both bind port 80 without conflict.

MNT namespace

Filesystem isolation

Own mount points. The container's root filesystem (the image) doesn't pollute or see the host's.

UTS namespace

Hostname isolation

Container can have its own hostname, independent of the host machine's — that random hex string you see in a shell prompt.

IPC namespace

Inter-process isolation

Isolates shared memory segments and semaphores so containers can't interfere with each other's IPC.

USER namespace

UID/GID isolation

Root inside the container can map to an unprivileged UID on the host — a key rootless-container security control.

unshare / nsenter
hands-on

The tools that prove namespaces aren't Docker magic — you can create and enter them with plain Linux utilities.

$ unshare --pid --fork --mount-proc /bin/bash
# launches bash in a NEW pid namespace — inside, "ps aux" shows almost nothing
$ ls -la /proc/<container_pid>/ns/
# every running process exposes links to the namespaces it belongs to
$ nsenter -t <pid> -n -p bash
# enter the network + pid namespace of an existing process — like "docker exec" by hand
INTERVIEW ANGLE → "What does docker exec actually do under the hood?" It calls setns() to join the existing namespaces of the target container's PID 1 — that's literally what nsenter demonstrates manually. There's no separate "container runtime magic"; it's the same syscall.
06 · Containers

Control Groups (cgroups)

Namespaces give isolation — what you can see. cgroups give resource limits — how much you can use. Together, that's the whole container model. This is also exactly what a Kubernetes resources.limits block configures.

Namespaces answer

  • What processes can I see?
  • What network interfaces exist?
  • What's my filesystem root?
  • What's my hostname?

cgroups answer

  • How much CPU can I use?
  • How much memory before OOM-kill?
  • How much disk I/O bandwidth?
  • How many PIDs can I spawn?
/sys/fs/cgroup
inspect

cgroups are exposed as a virtual filesystem — limits are just files you read and write.

$ cat /sys/fs/cgroup/memory.max
536870912   # 512 MiB — exactly what "docker run -m 512m" sets
$ cat /sys/fs/cgroup/cpu.max
50000 100000  # 50ms quota per 100ms period = 0.5 CPU cores
$ docker run -m 512m --cpus="0.5" myapp
# this flag is just a friendly wrapper writing to those exact files
WHY IT MATTERS → A container OOM-killed by the kernel isn't a Docker error — it's the memory cgroup hitting its limit and the kernel's OOM killer terminating the process. docker inspect showing OOMKilled: true and a Kubernetes pod status of OOMKilled both trace back to this exact mechanism. Check dmesg | grep -i oom on the host to confirm.
controllerlimits
cpuCPU shares and hard quota/period bandwidth limits
memoryMax memory + swap, triggers OOM kill when exceeded
io (blkio)Disk read/write throughput and IOPS throttling
pidsMax number of processes/threads — prevents fork-bomb style abuse
cpusetPin a process to specific CPU cores

Performance & Resource Monitoring

Before Prometheus and Grafana ever see a metric, it came from one of these commands reading /proc.

commanddoes
top / htopLive CPU, memory, per-process resource use
free -hMemory: used, free, cached, and swap — in human-readable units
vmstat 2System-wide CPU, memory, and I/O stats every 2 seconds
iostat -x 2Per-disk I/O throughput and latency — finds disk-bound bottlenecks

07 · Observability

Logs & journald

Before you ship logs to ELK or Loki, they live in journald and flat files. Knowing how to read them locally is what makes you fast during an incident, not just during normal ops.

journalctl
systemd logs
$ journalctl -u nginx -f
# follow live logs for the nginx unit
$ journalctl --since "10 min ago" -p err
# everything at error priority or worse, last 10 minutes
$ journalctl -b -1
# logs from the PREVIOUS boot — vital after an unexpected reboot
$ journalctl --disk-usage
# how much disk the journal itself is consuming
commanddoes
tail -f /var/log/syslogFollow a plain-text log file as it grows
less +F file.logtail -f behavior inside less — Ctrl+C then up/down to scroll back
logrotate -d /etc/logrotate.confDry-run log rotation rules — confirms what would happen before it runs for real

docker logs -f --tail 100 containerContainer logs, following, last 100 lines
